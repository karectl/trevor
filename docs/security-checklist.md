# trevor Security Checklist

Pre-deployment security review for the trevor egress/airlock microservice.

---

## Authentication & authorisation

- [ ] `DEV_AUTH_BYPASS=false` in all non-local environments
- [ ] `KEYCLOAK_URL` points to the production Keycloak realm (browser-facing)
- [ ] `KEYCLOAK_INTERNAL_URL` set for in-cluster OIDC calls (avoids loopback via ingress)
- [ ] JWT validation uses `python-jose` — verify `iss`, `aud`, `exp` claims on every request
- [ ] `tre_admin` realm role is checked from the JWT on every request (not cached in DB)
- [ ] Researcher/checker role conflict enforced at membership creation (`validate_no_role_conflict`)
- [ ] Submitter cannot review their own request (checked in `routers/reviews.py`)
- [ ] Two distinct reviewers required before approval (checked in `routers/reviews.py`)

## Secrets management

- [ ] All secrets in Kubernetes `Secret` objects — never in Helm values files or environment variable plaintext in manifests
- [ ] `SECRET_KEY` is a cryptographically random 32-byte value (see runbook)
- [ ] S3 credentials are not exposed to researchers (C-02 — trevor proxies all storage)
- [ ] SMTP credentials in `trevor-smtp` secret, not in env
- [ ] Rotate secrets on suspected compromise; rolling restart required after rotation

## Transport security

- [ ] TLS terminated at the ingress (NGINX or equivalent)
- [ ] `cert-manager` managing TLS certificate with auto-renewal
- [ ] `proxy-read-timeout` set high enough for SSE and large uploads (300s)
- [ ] Internal service-to-service traffic within the cluster (app ↔ DB, app ↔ Redis) not exposed outside

## CSRF protection

- [ ] All mutating HTML form submissions include a CSRF token (generated by `trevor.csrf.generate_csrf_token`)
- [ ] CSRF token validated on `POST`/`PATCH`/`DELETE` form endpoints
- [ ] Session cookie is `HttpOnly`, `SameSite=Lax`, `Secure` in production

## Rate limiting

- [ ] `slowapi` rate limiter is active (configured in `app.py`)
- [ ] Review the per-endpoint limits in `routers/` — tighten for sensitive endpoints (login, upload)
- [ ] Consider cluster-level rate limiting at the ingress for DDoS protection

## Container security

- [ ] Dockerfile uses a non-root user (see `Dockerfile` — user `trevor`, UID 1000)
- [ ] Base image is minimal (python-slim); no shell utilities in final stage
- [ ] Image is scanned with Trivy in CI (`trivy image --exit-code 1 --severity HIGH,CRITICAL`)
- [ ] `runAsNonRoot: true` and `readOnlyRootFilesystem: true` in pod `securityContext`
- [ ] Drop all Linux capabilities (`capabilities.drop: [ALL]`)
- [ ] No `hostPath` volume mounts; no `privileged: true`

## Network policy

- [ ] `networkPolicy.enabled: true` in production Helm values
- [ ] App pods only accept ingress from the ingress controller namespace
- [ ] Worker pods only accept egress to PostgreSQL, Redis, S3, Keycloak, SMTP, and LLM endpoints
- [ ] No unrestricted egress from app or worker pods

## Data integrity

- [ ] `OutputObject` SHA-256 checksum verified at every state transition (C-03)
- [ ] `AuditEvent` table has no `UPDATE`/`DELETE` grants for the application DB user (C-05)
- [ ] trevor never writes Kubernetes CRDs — only reads (C-06)
- [ ] RO-Crate assembled only at `RELEASED` state (C-11)

## Dependency security

- [ ] `uv lock` is committed and reproduced in CI (`uv sync --frozen`)
- [ ] Dependabot or Renovate configured for automated dependency updates
- [ ] `uv audit` (or `pip-audit`) run in CI to detect known CVEs
- [ ] Pre-commit hooks (`prek`) run on every commit: large-file check, private-key detection, ruff lint

## Operational

- [ ] Log level set to `INFO` (not `DEBUG`) in production — `DEBUG` may log sensitive headers
- [ ] JSON log format (`LOG_FORMAT=json`) for structured ingestion into Loki/ELK
- [ ] No stack traces exposed in HTTP responses (FastAPI default exception handler sanitises these)
- [ ] Health endpoint (`/health`) accessible without auth for liveness/readiness probes
- [ ] Readiness probe waits for DB connection pool to be healthy before accepting traffic
